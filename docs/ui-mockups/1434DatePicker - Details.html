Description

A form control combining a text input with an icon to open a calendar pop-up. It allows users to select a single date or a date range.

  • The design is desktop-first. On mobile viewports, the native `` is recommended for optimal user experience, which would be handled by a different component or logic.

The calendar grid should be navigable with arrow keys. 'Enter' to select a date. 'Escape' to close the calendar. (Note: Full keyboard logic requires JavaScript).

Implementation Notes

Embedded Css Architecture

All styles are embedded, using design system tokens as CSS custom properties and a BEM-like naming convention (.datepicker__element--modifier).

Component Organization

The showcase demonstrates single date and date range pickers, including various states like disabled and error. An 'is-open' class is used for demonstration, while `:focus-within` provides a CSS-only interactive example.

Responsive Design

The component is designed for desktop viewports. A note is included about using native pickers for mobile.

Accessibility Compliance

The static structure is built with accessibility in mind (semantic HTML, planned ARIA roles). Full compliance requires JavaScript for focus management and dynamic ARIA state updates.

Browser Support

Supports modern browsers. The calendar uses CSS Grid for layout.

Usage Guidelines

When To Use

Use for any form field that requires a user to select a specific date or a range of dates.

Content Guidelines

The placeholder text should be instructive, e.g., 'Select a date' or 'YYYY-MM-DD'.

Placement Rules

Typically used within forms, alongside other input controls.

Customization Options

  • The component can be adapted for date range selection by applying `.is-start-date`, `.is-end-date`, and `.is-in-range` classes to the day elements.

Testing Recommendations

Browser Testing

Test in Chrome, Firefox, and Edge.

Device Testing

Verify usability on desktop and tablet devices.

Accessibility Testing

  • Keyboard navigation testing (requires JS implementation).
  • Screen reader testing to ensure all elements are announced correctly.
  • Focus trapping verification within the calendar modal (requires JS).

Performance Testing

N/A for static component, but the JavaScript logic for rendering should be optimized.
