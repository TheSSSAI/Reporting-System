Description

A modal dialog is an overlay that requires user interaction. It disables the main content until the user explicitly interacts with the modal, either by completing an action or dismissing it. It's used for critical information, confirmations, or self-contained forms.

Design System Alignment

Utilizes design system tokens for color, spacing, typography, shadows, and border-radius to ensure visual consistency with other components.

Responsive Specifications

Mobile Adaptations

  • On viewports below 600px, the modal becomes a full-screen view to maximize content readability and usability on small devices.
  • Border radius is removed on mobile for a seamless full-screen appearance.

Tablet Adaptations

  • Maintains the standard centered dialog layout as on desktop.

Desktop Enhancements

  • Centered on the screen with a backdrop overlay to focus user attention.
  • Subtle scale and fade-in animations for a smooth appearance.
  • Constrained maximum width to ensure readability of content.

Accessibility Features

Keyboard Support

Full keyboard navigation is expected via JavaScript. The 'Escape' key should close the modal. Focus must be trapped within the modal when open and returned to the trigger element on close.

Screen Reader Support

Uses `role='dialog'`, `aria-modal='true'`, `aria-labelledby` pointing to the title, and `aria-describedby` pointing to the body content. Close button has an `aria-label`.

Focus Management

Visible focus indicators on all interactive elements (buttons, inputs). JavaScript is required to manage focus trapping.

Aria Attributes

  • role='dialog'
  • aria-modal='true'
  • aria-labelledby
  • aria-describedby
  • aria-label for close button

Color Contrast

All text and interactive elements meet WCAG 2.1 AA contrast ratio requirements.

Implementation Notes

Embedded Css Architecture

All styles are embedded, using design system tokens as CSS custom properties and a BEM-style naming convention (`modal-dialog`, `modal__header`, etc.).

Component Organization

The HTML file showcases three distinct variants of the modal. A CSS-only `:checked` pseudo-class hack is used to demonstrate the open/closed states without JavaScript.

Responsive Design

A single breakpoint at 600px handles the transition from a centered dialog to a full-screen mobile view.

Accessibility Compliance

The static HTML/CSS is structured for accessibility. Full compliance requires JavaScript for focus management and keyboard interactions (e.g., Esc key).

Browser Support

Designed for modern browsers. The CSS-only toggle is for demonstration; a production implementation would use JavaScript to manage state.

Usage Guidelines

When To Use

Use for critical user tasks that must be completed before returning to the main application flow, such as confirming destructive actions, displaying important information, or hosting short, self-contained forms.

Content Guidelines

Keep content concise. Titles should be short and direct. Body text should be brief. If extensive content is needed, consider a dedicated page instead.

Placement Rules

Modals are triggered by user actions (e.g., button clicks) and are always centered over the application viewport.

Customization Options

  • The modal body content is fully customizable.
  • The modal footer can contain any combination of action buttons (e.g., Primary, Secondary, Danger).

Testing Recommendations

Browser Testing

Test in Chrome, Firefox, Safari, Edge.

Device Testing

Test on desktop and mobile viewports to verify the responsive behavior.

Accessibility Testing

  • Verify focus is trapped inside the open modal.
  • Verify the 'Escape' key closes the modal.
  • Verify focus returns to the triggering element on close.
  • Test with a screen reader (NVDA/JAWS/VoiceOver) to ensure roles and labels are announced correctly.

Performance Testing

Ensure animations are smooth and do not cause layout shifts.
